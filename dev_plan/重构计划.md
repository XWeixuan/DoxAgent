# Task 内 ReAct Memory 重构计划

本计划以 `dev_plan/ReAct_Memory_Restructuring.md` 为设计依据。目标是一次性替换单个 `AgentTask` 内、多个 ReAct loop 之间的 memory 机制；不做新旧双轨，不保留 `Scratchpad` legacy。

## 1. 当前实现结论

当前主链路是：

```text
workflow 创建 AgentTask
-> ModelGatewayAgentRunner 选择 react runtime
-> ReActAgentHarness 为 task 新建 Scratchpad
-> 每轮重新组装 prompt
-> AgentResult.react_audit 写入 Working Memory
```

主要问题集中在 `src/doxagent/agents/runtime/react.py`：

- `Scratchpad.entries` 同时保存审计事件、tool observation 和下一轮上下文；
- `microcompact()` 会改写旧 observation，`append_compaction_summary()` 会删除 tool/delegation entry；
- 下一轮 prompt 直接依赖 `task_ledger`、`compacted_evidence_summary`、`market_evidence_snapshot` 和 `recent_trajectory`；
- full compaction 使用固定 12,000 token 阈值和通用 LLM 摘要，不是基于真实模型上下文预算；
- 最终 `react_audit` 又被 workflow 写入 Blackboard Working Memory，导致 task 内 memory 与跨节点审计投影容易混淆。

`ContextBuilder`、workflow task input 和 Blackboard 负责的是 task 外部输入及跨节点状态。它们不是本次要替换的 task 内 memory。

## 2. 重构边界

### 2.1 本次重构范围

凡是由 `ReActAgentHarness` 执行、且一个 `AgentTask` 内可能运行多个 loop 的任务，都切换到新 memory：

- Immutable Task Event Log；
- Raw Tool Result Store；
- Observation Parser / Block Store / Pointer；
- Working Synthesis、Research Agenda、Retained Observations、Plan、最近两轮 Reasoning Summary；
- Active Context View；
- Fresh Observation 生命周期；
- `read_observation` task 内只读能力；
- context budget、Micro Context Maintenance、Full Compaction；
- pre-final research challenge；
- 新的 task memory 审计输出。

范围按 runtime 机制判断，不按 Document1/2/3 节点分别实现。初始化 workflow 的 ReAct task，以及 Persistent Runtime 中实际走 ReAct 的 O3/A2 task，使用同一套实现。

### 2.2 必须完全移除的 legacy

开发完成后不再保留以下实现或兼容分支：

- `Scratchpad` 类及其 `entries` 混合存储；
- `microcompact()`、`append_compaction_summary()` 和旧 `_compact_if_needed()`；
- 旧 observation preview/marker 式 `_compact_tool_observation()`；
- `task_ledger` / `task_ledger_updates`；
- `recent_trajectory`、`compacted_evidence_summary` prompt 字段；
- Scratchpad 自管的 `compacted_summaries`、`market_evidence_snapshots`；
- 固定 `compaction_token_threshold`；
- 依赖上述字段的 legacy 测试和 fallback 判断。

现有 tool 输出里的 `market_evidence_snapshot` 可以继续作为原始 ToolResult / EvidenceRef 内容存在，但不再作为 Scratchpad 特例 memory；由 Observation Policy 按 `recomputable` 数据处理。

`AgentResult.payload.react_audit` 这个 workflow 外层审计入口暂时保留，避免修改 Blackboard 写入边界；但其内部 schema 将原子替换为新 Event Log / Memory / Observation / Context Budget 审计结构，不兼容旧 `entries` schema。

### 2.3 本次暂不改动

- Blackboard 的长期业务状态、文档桶、commit/objection/delegation 模型；
- Document1 / Document2 / Document3 的节点顺序、跨节点 handoff 和 task input 裁剪策略；
- `AgentTask`、业务 `AgentResult`、各节点 final output schema；
- 现有 `EvidenceRef` 业务契约；
- tool provider/client 的业务取数逻辑；
- agent 私有长期记忆、跨 run 经验检索；
- W1 / W2 等 `single_shot` worker；
- 数据库表结构和历史 Working Memory 数据迁移；
- 进程崩溃后的 task 中途续跑。新 Event Log 先支持 task 内 reducer 重放和结果审计，持久化断点续跑另行设计。

`ContextBuilder`、`PromptAssembler`、workflow audit 只做新审计/上下文接口的必要适配，不借本次重构改变跨节点业务语义。

### 2.4 切换约束

不做 feature flag、双写或 legacy hydrate。部署切换前必须结束或放弃正在运行的 ReAct task；切换后的新 task 只使用新实现。历史 `react_audit` 仅作为只读历史记录，不用于恢复新的 Task Memory State。

## 3. 目标代码结构

将 `react.py` 中的 memory 责任拆出独立 package，例如：

```text
src/doxagent/agents/runtime/memory/
├── events.py          # append-only Event Log 与事件模型
├── observations.py    # Raw Store、Block、pointer、parser adapter
├── policies.py        # inline / indexed / recomputable policy
├── state.py           # materialized Task Memory 与 reducer
├── context.py         # Active Context 与 token budget
└── protocol.py        # memory update / maintenance action 解析
```

`react.py` 只保留 loop 编排、模型调用、工具/委托执行和 final 交付。Event Log 的 tool-result event 通过 `raw_result_id` 指向 Raw Store；完整 ToolResult 只存一份，但仍可由 Event Log 精确恢复。其他视图也只使用 ref，避免同一原文在审计 payload 中重复膨胀。

## 4. 开发步骤

### 第一步：建立基线并做真实 ToolResult Profiling

1. 固定当前 AgentTask/AgentResult、final schema、工具限流、delegation 和失败 fallback 的行为基线。
2. 从真实 LangSmith traces 采样常规、大结果、空结果和 partial result。
3. 将脱敏样本保存为 fixture，记录自然切块单位、父级上下文、原生 ID、EvidenceRef 关联和建议 policy。
4. 先完成通用 JSON/list/text/table/time-series adapter 映射，再决定 DoxAtlas 专用 hierarchy override。

没有真实 profile 的工具不得仅凭名称写专用 parser。

### 第二步：实现不可变 Event Log 和 Task Memory reducer

1. 定义 append-only task events，并覆盖 action、tool、skill、delegation、warning、memory update、compaction、final/failure。
2. 定义 Working Synthesis、Research Agenda、Retained Observations、Plan 和最近两轮 Reasoning Summary。
3. 使用 reducer 从 Event Log 生成当前 materialized state；测试重放结果一致。
4. 保留 tool 次数、连续调用次数、batch 次数等 runtime guard，但不把它们混入 agent memory。

### 第三步：实现 Observation Data Layer

1. Harness 为每次工具调用分配 task 内稳定 `tool_call_id`。
2. 完整 ToolResult 原样进入 Raw Store，不在写入时截断。
3. 确定性 parser 生成 Observation Blocks、Context Envelope、hash 和 agent 可见 ref。
4. 实现 pointer validation 与 `read_observation`；无效 ref 只告警，不阻塞正常 tool/final action。
5. 在 `ToolDescriptor` 或独立 registry 中配置 `inline/indexed/recomputable` policy。

### 第四步：接入 Agent Action Protocol 和 memory prompt

1. 新增 `synthesis_update`、`research_update`、`retain_observations` 的软解析和应用事件。
2. 移除 `task_ledger_updates`。
3. 将现有 `prompts/workflows/memory.md` 接入所有 ReAct task，但不注入 W1/W2 single-shot task。
4. Active Context 固定按设计顺序输出 Research Frame、Memory State、最近两轮 Reasoning Summary、Retained/Fresh Observations 和 warnings。

### 第五步：一次性替换 ReActHarness 主循环

1. 以新 Task Runtime State 替换 `Scratchpad(task)`。
2. tool 返回后依次写 Event Log、Raw Store、Observation Blocks，再生成只展示一轮的 Fresh Observation。
3. 每轮 action 后应用增量 memory update；未 retain 的旧 Fresh Observation 退出 Active Context，但原文不删除。
4. skill、delegation、tool limit、max_steps 和现有 final schema 校验继续保留。
5. 同一提交中删除旧 Scratchpad、旧 prompt 字段和旧 compaction 路径，不留 fallback 到 legacy 的入口。

### 第六步：实现 Context Budget 与新 Compaction

1. 配置模型 context window、reserved output、tool schema 和 safety reserve，按实际待发送 prompt 估算输入。
2. 超过 75% 先做确定性 Micro Context Maintenance，不修改 Event Log、Raw Store 或 Task Memory。
3. Micro 后超过 85% 才由同一个主 agent 在同一 loop 执行 Full Compaction maintenance action。
4. 保护尚未首次消费的 Fresh Observation；Full Compaction 只能维护 Synthesis、Agenda、Plan 和 retained 加载状态。
5. maintenance 最多重试一次；仍超限时只把大块 retained 原文降为 `index_only`，不 DROP、不改原文。

### 第七步：补齐 final、审计和 workflow 边界适配

1. 研究型 task 首次准备完成时执行最多一次 pre-final challenge。
2. final 前重新加载与有效 Synthesis 直接关联的 observation 原文。
3. 新 `react_audit` 输出 Event Log、当前 Memory State、Observation/Context Budget 统计和 warning；Raw ToolResult 不重复嵌入多个位置。
4. 更新 `_write_working_memory()`、`_with_tool_usage_audit()` 和 `ContextBuilder` 的 compact audit projection，使下游节点默认看不到完整 Event Log/Raw Store。
5. max-steps/failure fallback 改为读取新 state/store，不再读取 `compacted_summaries` 或 legacy entries。

### 第八步：测试、删除确认和真实 smoke

1. 单测：event append-only、reducer、pointer 1:1恢复、结构切块、policy、memory update、Fresh Observation 生命周期。
2. 集成测试：多 loop、并行 tools、delegation、invalid ref、Full Compaction、final reload、max-steps/failure。
3. 回归测试：初始化 workflow 与 O3/A2 ReAct task 的 final schema、EvidenceRef、tool limit 和 Working Memory 写入不变。
4. 静态删除确认：仓库中不再出现 `Scratchpad`、`microcompact`、`recent_trajectory`、`compacted_evidence_summary`、`task_ledger_updates` 等 legacy runtime 依赖。
5. 运行 focused pytest、完整相关测试、ruff、mypy；再用一条真实 ReAct trace 验证 pointer、memory 增量、context budget 和 final audit。

## 5. 验收条件

- Event Log、Raw ToolResult、Observation Block 永不被 compaction 改写或删除；
- agent 正常 prompt 看不到完整 Event Log 和全部历史 ToolResult；
- ref 可以稳定、精确恢复原文/原值并关联 EvidenceRef；
- agent 始终看到当前有效 Synthesis、active/deferred Agenda、Plan、最近两轮 Reasoning Summary 和 loaded retained 原文；
- Fresh Observation 至少完整展示一轮，之后按 retain 决定是否继续加载；
- Full Compaction 只改变 materialized memory 和 Active Context，不改变历史事实；
- final 主要基于 Working Synthesis，并在需要精确事实时读取原始 observation；
- 新旧 memory 不并存，legacy 代码和 legacy 测试全部移除；
- workflow 节点、Blackboard 业务状态、EvidenceRef 和 final output contract 不因本次重构改变。

本轮只完成计划，不修改实现代码。下一轮严格按以上顺序执行。
